# Design Thinking Session: AgentsGate — 使用者上傳體驗

**Date:** 2026-03-03
**Facilitator:** Jackson
**Design Challenge:** 如何讓開發者「上傳一次」，平台自動識別 Skill/Agent 類型並產生所有 CLI 的適配文件

---

## 🎯 Design Challenge

開發者已在本地用不同 CLI 建立了各式 Skill 或 Agent，但因各 CLI 格式碎片化，他們無法輕鬆將成果分享給使用不同工具的人。AgentsGate 需要設計一個上傳體驗，讓上傳者「最少一次」即可讓作品適配所有 CLI。

### 三層摩擦點

1. **識別層**：平台如何辨識上傳的是哪種類型？（Skill vs Agent、來自哪個 CLI？）
2. **補全層**：每個 CLI 需要的欄位不完全相同，缺少的資訊怎麼補？
3. **轉換層**：格式、語法、路徑引用方式都不同，如何自動生成各 CLI 的適配檔？

### 關鍵設計決策：上傳前的強制分叉

上傳流程必須先引導使用者區別 **Skill** 還是 **Agent**，因為兩者的適配邏輯、支援 CLI、必填欄位與審核方式完全不同：

| 面向 | Skill | Agent |
|------|-------|-------|
| 開放標準 | AgentSkills SKILL.md（Claude + Copilot 共用） | 各 CLI 各自為政，無統一標準 |
| 支援 CLI | Claude Code、GitHub Copilot、Gemini、Kiro、Cline | 目前主要 Copilot `.agent.md` |
| 核心欄位 | `name` + `description` | `description` + `tools[]` + 角色人格 |
| 安全審核 | 中 | 高（角色/人格需額外審查） |

**類型可修改原則**：若使用者選錯類型，允許事後修改並重新觸發適配文件生成，不需要重新上傳原始內容。

### 成功定義

上傳者只需提交「核心內容 + 少量必填欄位」，平台自動偵測類型並生成所有 CLI 的適配文件，不需要上傳者了解各 CLI 的格式規範。類型選錯可修正，平台重新生成而不懲罰使用者。

## 👥 EMPATHIZE: Understanding Users

### User Insights

**來源：與主要上傳者 Jackson 的訪談（本人即典型上傳者）**

1. **環境依賴問題**：Skill 在本地環境可以運行，但換到其他 PC 不一定能跑。尤其當 Skill 需要呼叫本地服務或外部 API 時，防火牆未開通會導致失敗。上傳時需要讓上傳者清楚聲明環境需求，讓下載者知道前置條件。

2. **安裝路徑混亂**：各家 CLI 的 Skill 放置路徑完全不同，下載後使用者不知道該把檔案放在哪裡，需要靠猜測或翻文件。

3. **可移植性焦慮**：上傳者最擔心的事是「這個 Skill 到別人的環境無法適配」，這種擔心在發布前無法被驗證或緩解。

4. **Skill vs Agent 邊界模糊**：上傳者希望在選擇類型前，能看到清楚的差異對比和警示說明，讓自己想清楚再決定，而不是事後才發現選錯。

### Key Observations

**四層摩擦點（新增第四層）：**

| 層次 | 問題 | 影響對象 |
|------|------|---------|
| 識別層 | 平台如何判斷 Skill vs Agent、來自哪個 CLI | 平台 |
| 補全層 | 各 CLI 必填欄位不同，缺少資訊怎麼補 | 平台 + 上傳者 |
| 轉換層 | 格式/語法/路徑引用方式不同，如何自動轉換 | 平台 |
| **可移植性層**（新） | Skill 內容本身是否能在其他環境執行，與格式無關 | 上傳者必須自行聲明 |

**關鍵觀察：**
- 「格式適配」和「內容可移植」是兩個不同的問題，平台只能解決前者，後者需要上傳者負責聲明
- 環境依賴資訊（需要哪些服務、網路條件、前置套件）目前沒有標準化欄位
- 下載後的安裝路徑問題 → 已由安裝腳本（install.ps1/install.sh）解決，但上傳者不一定知道這件事，會重複焦慮
- 類型選擇是高認知負擔的決策點，需要「說明 + 範例 + 對比」三位一體的引導

### Empathy Map Summary

**上傳者在「準備上傳 Skill」這個情境下：**

| 維度 | 內容 |
|------|------|
| **說（SAY）** | 「這個 Skill 我自己用起來很順」、「我不確定別人能不能用」、「我不確定這算 Skill 還是 Agent」 |
| **想（THINK）** | 「萬一別人下載後跑不起來怎麼辦」、「我要填哪些欄位？」、「其他 CLI 的格式我不熟，會不會轉錯」 |
| **做（DO）** | 把 Skill 放在自己習慣的資料夾、測試自己的 CLI、沒有測試其他 CLI 的相容性 |
| **感（FEEL）** | 不確定感（我的 Skill 夠好嗎？）、焦慮（別人的環境跟我不一樣）、困惑（類型怎麼分？） |

**核心情緒：焦慮於「無法控制下載後的體驗」**

**補充洞察（環境聲明欄位設計）：**
- 環境需求需要「結構化必填欄位 + 自由描述文字」雙軌並行
- 結構化欄位（如 checkbox）確保平台可程式化處理（需要網路？需要本地服務？需要 MCP Server？）
- 自由描述文字讓上傳者補充無法被結構化的特殊需求
- **平台必須嚴格規範上傳格式**，避免每個人丟出格式不統一的內容，導致後續自動轉換和 AI 審核失效
- 核心矛盾：太自由 → 平台無法處理；太嚴格 → 上傳門檻高、嚇跑貢獻者
- 解法方向：結構化必填（Platform 強制）+ 自由補充（User 選填），兩層分開

## 🎨 DEFINE: Frame the Problem

### Point of View Statement

**主要上傳者 POV：**
> 技術型開發者（已有本地 CLI Skill）**需要**一個有明確格式規範的上傳流程，引導他們完成類型選擇、環境依賴聲明、與內容標準化；**因為**他們的 Skill 雖然在本地可用，但缺乏可移植性宣告與格式一致性，導致下載者無法判斷相容性、平台無法自動轉換適配文件。

**下載者 POV（延伸）：**
> 使用各種 CLI 工具的開發者**需要**清楚的 CLI 專屬載入引導文件；**因為**各 CLI 載入 Skill 的機制（路徑、格式、指令）完全不同，光有安裝腳本還不夠，他們需要知道「下載後在我的 CLI 裡要怎麼使用」。

### How Might We Questions

| # | HMW 問題 | 對應洞察 |
|---|---------|---------|
| 1 | HMW 幫助上傳者在 30 秒內判斷自己的東西是 Skill 還是 Agent，同時不讓他們覺得被考試？ | 類型邊界模糊 |
| 2 | HMW 讓環境依賴聲明變得輕鬆，而不是又一個讓人頭痛的表單？ | 環境依賴不透明 |
| 3 | HMW 在強制格式規範的同時，保留上傳者補充說明的空間？ | 格式自由 vs 平台可處理性 |
| 4 | HMW 讓每個 CLI 的使用者在下載後，立刻知道怎麼在自己的工具裡啟動這個 Skill？ | 安裝路徑混亂 + CLI 引導文件缺失 |
| 5 | HMW 讓上傳者在發布前對「可移植性」有信心，而不是帶著焦慮送出去？ | 可移植性焦慮 |

### Key Insights

**三個核心洞察：**

1. **平台服務兩端，但目前設計只想到上傳者**
   上傳流程設計了格式轉換，但沒有為下載者設計「各 CLI 載入引導文件」。每個 Skill 應自動產生一份對應各 CLI 的 Quick Start 說明（例如：Claude Code 怎麼用、GitHub Copilot 怎麼用），這份說明應內嵌在下載套件中，也顯示在平台詳細頁。

2. **「格式一致性」是平台生命線，必須從上傳入口就守住**
   若上傳格式不統一，AI 審核和自動轉換都會失效。上傳表單必須是「結構化必填欄位」驅動，而非讓使用者自由貼上任何格式。平台應提供清楚的欄位說明與範例，降低填寫門檻。

3. **上傳者最大的痛點是「不確定感」，不是技術能力不足**
   技術型開發者知道怎麼做 Skill，但不知道平台怎麼處理、轉換後對不對、別人能不能用。平台需要在上傳流程中提供即時回饋（預覽生成結果、驗證格式、顯示相容 CLI 清單），讓上傳者在送審前就有把握。

## 💡 IDEATE: Generate Solutions

### Selected Methods

- **Brainstorming**：針對每個 HMW 快速生成點子，不評判
- **Crazy 8s**：強迫找極端或意外的解法突破慣性
- **SCAMPER**：從現有上傳流程變形出新設計角度

### Generated Ideas

**HMW1 — 30 秒判斷 Skill vs Agent：**
1. 互動式決策樹（「你的東西有角色人格嗎？」→ Agent；「只執行特定任務？」→ Skill）
2. 左右對比卡：Skill 範例 vs Agent 範例，附上 BMAD 真實案例
3. 貼上內容後 AI 自動建議類型（分析 frontmatter 和內容關鍵字）
4. 只問一個問題：「下載後會以 `/指令` 方式呼叫，還是以角色對話方式使用？」
5. 「我不確定」選項 → 展開詳細比較頁面

**HMW2 — 環境依賴聲明輕鬆化：**
6. 結構化 Checklist（需要網路連線 / 需要 MCP Server / 需要本地服務 / 需要 Docker）
7. 預設全部無需依賴，只有勾選才出現補充欄位（降低視覺噪音）
8. 平台分析上傳內容自動偵測疑似依賴（例如偵測到 MCP config 就自動勾選）
9. Compatibility 欄位自動生成文字預覽（根據勾選結果產生一句話說明）

**HMW3 — 格式規範但不嚇跑人：**
10. 表單驅動，不需要手寫 YAML，平台幫你生成 SKILL.md
11. 貼入現有 SKILL.md → 平台自動解析並填入對應欄位（反向匯入）
12. 即時格式驗證（類似 IDE 的即時紅線提示，填完就知道有沒有問題）
13. 分步 Wizard（一次一個欄位，不要一次看到整張表單）
14. 每個欄位旁邊有「看範例」Tooltip，展示好的和不好的範例對比
15. 草稿自動儲存，關掉可以繼續填

**HMW4 — CLI 載入引導文件（套件內 + 平台頁同步）：**
16. 每個 CLI 子目錄自動生成 `QUICKSTART.md`（安裝後放在哪、怎麼呼叫）
17. 平台詳細頁有 CLI Tabs（Claude Code / Copilot / Gemini / Kiro），點選顯示該 CLI 的載入指令
18. 安裝腳本（install.ps1/install.sh）跑完後在終端印出 Quick Start 摘要
19. 平台提供「複製指令」按鈕（一鍵複製 CLI 載入語法）

**HMW5 — 發布前的信心：**
20. 送審前「預覽」頁：列出平台將生成的所有 CLI 適配文件內容（可逐一展開查看）
21. 相容性清單：顯示每個 CLI 的相容狀態（✅ 支援 / ❌ 不支援 + 原因）
22. AI 格式預審：送審前先跑格式驗證，列出問題讓上傳者修正後再送
23. 「模擬下載者視角」按鈕：讓上傳者看到下載者在平台頁面上會看到什麼

### Top Concepts

| Concept | 核心想法 | 解決的 HMW |
|---------|---------|-----------|
| **A. 智慧上傳精靈（Smart Upload Wizard）** | 分步引導 + 支援貼入現有 SKILL.md 自動解析 + 即時格式驗證 + 送審前預覽適配結果 | HMW 2, 3, 5 |
| **B. 類型選擇器（Type Decision Card）** | 互動對比卡 + AI 建議類型 + 可事後修改並重新生成所有 CLI 適配文件 | HMW 1 |
| **C. CLI 引導文件自動生成系統** | 每個子目錄自動生成 QUICKSTART.md + 平台 CLI Tabs + 安裝腳本尾聲摘要 + 一鍵複製指令 | HMW 4 |

**三個 Concept 整合為一套完整的上傳體驗流程**：Type Decision Card（入口）→ Smart Upload Wizard（填寫）→ CLI 引導文件自動生成（輸出）

### Key Learnings

**已驗證、可直接進入設計實作的決策：**

1. 上傳流程必須從「類型選擇」開始，且類型選擇頁需要有真實範例對比

4. 類型選錯允許修改並重新生成，不懲罰使用者
